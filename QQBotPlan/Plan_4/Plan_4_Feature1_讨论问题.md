## 一、向量模型选型

### Q1. 用 Gemini 云端 Embedding 还是本地模型？

**选项 A：Gemini `text-embedding-004`（云端）**
- 优点：768 维向量、多语言支持好（中文优秀）、精度高
- 缺点：每次增删 Memory 都要调 API（虽然只要 $0.15/百万 token 很便宜）、需要网络
- 延迟：约 50-150ms/次

**选项 B：本地模型（如 `m3e-base` 或 `bge-base-zh`）**
- 优点：完全离线、零成本、响应更快（10-30ms）
- 缺点：需要额外下载模型文件（约 400MB）、中文效果可能略逊 Gemini
- 需要安装 `sentence-transformers` + `torch`（约 2GB 磁盘）

**选项 C：混合方案**
- Memory 数量少 → 用 Gemini API（精度优先）
- QQ_data 历史量大 → 用本地模型（速度和成本优先）

> 你更倾向哪个？如果选本地模型，你能接受额外 2GB 左右的磁盘占用吗？

### Q2. 向量维度偏好？

这个影响搜索精度和速度。一般有这几档：
- 384 维：速度最快，精度一般（适合百万级数据量）
- 768 维：主流平衡（Gemini embedding 默认就是这个）
- 1024 维：精度最高，但搜索慢一点

> 一般 768 维是最佳选择，除非你有特别的想法？

## 二、向量存储引擎选型

### Q3. 存储引擎选哪个？

**选项 A：FAISS（Facebook AI 出品）**
- 纯内存运行，速度极快（微秒级搜索）
- 10 万条数据也只占约 300MB 内存
- 但关闭程序后需要从文件重新加载
- 不支持带条件过滤（比如"只搜某个群的记忆"）

**选项 B：ChromaDB**
- 支持持久化存储（SQLite 后端，自动保存）
- 支持 metadata 过滤（可以按群号、时间范围等条件筛选）
- 比 FAISS 慢一些，但更方便
- 额外依赖：`pip install chromadb`（约 200MB）

**选项 C：SQLite-VSS（基于现有 SQLite）**
- 复用我们已有的 SQLite 基础设施
- 轻量、不需要额外服务
- 但功能不如 ChromaDB 丰富

> 举例区别：
> - 如果你只想"搜全局最相关的 5 条"，FAISS 足够
> - 如果你想"搜**这个群**中最相关的 5 条"，需要 ChromaDB 的 metadata 过滤
> - 你觉得需要按群/按用户/按时间范围过滤搜索结果吗？

## 三、数据源与索引策略

### Q4. 哪些数据需要向量化？

以下是可以向量化的数据类型，勾选你觉得需要的：

- [ ] **Memory 条目**：FlashLite 生成的用户画像和记忆（目前约几十条/群）
- [ ] **Knowledge 内容**：FlashLite 压缩后的知识快照

> 不同数据类型的向量化粒度不同。比如 Memory 是"一条一条"的，而 QQ 聊天记录可能需要"滑动窗口"切分（每 5 条合成一个片段向量化）

### Q5. 增量索引还是定期全量重建？

**选项 A：增量索引（实时）**
- 每当 Memory 新增/修改/删除时，立刻更新向量索引
- 实现复杂度稍高，但索引始终是最新的

**选项 B：定期全量重建（批量）**
- 比如每天凌晨或每 6 小时全量重建一次
- 实现简单，但有一个"索引滞后窗口"

**选项 C：混合**
- Memory 用增量（量小、变更频繁）
- QQ 聊天记录用定期重建（量大、增长稳定）

> 你对"最新的记忆要多快被搜索到"有要求吗？
> 比如老板娘 5 分钟前学到的新记忆，你希望下一次就能 Top-K 命中，还是隔几小时也行？

### Q6. QQ 聊天记录的切分粒度？

如果要对 QQ 历史聊天记录向量化（Q4 中选了的话），需要决定怎么切分：

**选项 A：逐条消息**
- 每条 QQ 消息都是一个独立向量
- 优点：精度最高，能定位到具体一条
- 缺点：量太大（10 万条消息 = 10 万个向量），搜索结果可能缺乏上下文

**选项 B：滑动窗口（推荐）**
- 每 5-10 条消息组成一个"对话片段"，相邻片段有 2-3 条重叠
- 优点：保留了对话上下文，搜索结果可读性好
- 缺点：需要清洗（比如过滤纯表情/图片消息）
- 举例：
  ```
  片段 1: [消息1, 消息2, 消息3, 消息4, 消息5]
  片段 2: [消息3, 消息4, 消息5, 消息6, 消息7]  ← 重叠2条
  片段 3: [消息5, 消息6, 消息7, 消息8, 消息9]
  ```

**选项 C：按话题自动分段**
- 用 FlashLite 或规则检测话题转换点，按"一段对话"为单位
- 质量最高，但实现最复杂

> 你偏好哪种？滑动窗口的话，窗口大小（5 条？10 条？）和重叠量（2 条？3 条？）怎么定？

## 四、三个应用场景的具体决策

### Q7. 应用场景 1（Memory 混合推荐）的融合策略？

两路并行推荐后，FlashLite 怎么做最终融合？

**选项 A：简单合并去重**
- 向量侧 Top-K1 + FlashLite 侧 Top-K2 → 去重后全部呈给 FlashLite → FlashLite 最终筛选
- 最简单，但 FlashLite 收到的候选量可能翻倍

**选项 B：加权排序**
- 向量侧有相似度分数，FlashLite 侧有重要度标签 → 综合打分后取 Top-N
- 需要设计打分公式

**选项 C：互补模式**
- 向量侧只推荐 FlashLite 侧**没推荐过的**（去重后取补集）
- 最大化两侧的互补性，避免重复推荐

> 你觉得哪种更合理？向量侧推荐几条比较合适（3？5？10？）

### Q8. 应用场景 3（联想机制）的注入方式？

这是你新提的想法，需要决定几个关键细节：

#### Q8a. 注入位置是哪里？

**选项 A：system_instruction 尾部**
- 在系统提示词末尾加一段"相关历史上下文"
- 优点：简单，主模型一定能看到
- 缺点：会影响 KV Cache hash（每次联想内容变化 → 缓存重建）

**选项 B：conversation 开头（第一条 user message 之前）**
- 作为"背景知识"插入到对话历史的最前面
- 优点：不影响 system_instruction 的 KV Cache
- 缺点：可能被长对话推出上下文窗口

**选项 C：当前 user message 的上方（紧贴用户消息）**
- 把联想内容作为"对话上下文"放在用户当前消息的正上方
- 优点：主模型读到时离用户消息很近，关联性强
- 缺点：需要修改 prompt 拼装逻辑

> 这个位置决定了联想内容对主模型的"影响力大小"和对缓存的影响。
> 你更倾向哪个？

#### Q8b. 每次注入多少条？

- 1-2 条：轻量联想，基本不增加成本
- 3-5 条：中等联想，可能每次多 500-1000 token
- 5-10 条：重度联想，成本显著增加

> 注意每条上下文片段可能有 100-300 token，5 条就是 500-1500 token/每次请求
> 你觉得每次花多少额外 token 在"联想"上是可接受的？

#### Q8c. 联想的触发条件？

**选项 A：每次主模型请求都联想**
- 任何时候用户跟老板娘说话，都先搜一圈相关历史
- 全覆盖，但成本最高（每次多一个 embedding 调用 + 向量搜索）

**选项 B：仅主动唤醒（@/关键词）时联想**
- FlashLite 同步/时间触发时不联想，只有用户主动找老板娘时才联想
- 减少不必要的联想

**选项 C：根据相似度阈值自动判断**
- 搜索后看 Top-1 的相似度，超过阈值（比如 0.75）才注入
- 如果没有足够相关的历史，就不注入（避免把不相关的东西塞进去干扰）

> 你觉得联想应该是"总是开着"还是"有相关内容时才触发"？

#### Q8d. 联想内容会不会干扰主模型？

这是一个需要你评估的风险：

**可能的好处**：
- "你之前不是说要减肥吗，怎么又吃了"——自然引用历史，增加亲密感
- 对话连贯性提升，老板娘显得"真的记得"

**可能的坏处**：
- 向量搜索给了一段不那么相关的历史 → 主模型强行关联 → 答非所问
  - 例：用户说"今天好冷"，向量搜到上个月"用户说冬至好冷"，主模型回复"上次冬至你也说冷来着"——但用户其实在说今天天气，不想翻旧账
- 大模型可能"过度联想"——把注入的历史上下文当成强指令来参考

> 你觉得这个风险可以接受吗？有没有什么缓解思路？
> 比如：联想片段加一个"仅供参考"标签？或者只在高相似度时才注入？

### Q10. 联想内容对 KV Cache 的影响？

我们讨论过 Knowledge 在 system_instruction 中导致 hash 变化 → 缓存重建的问题。
联想机制会让这个问题更突出：

- 如果联想内容放在 system_instruction → **每次请求的联想内容可能不同** → hash 每次都变 → 缓存几乎永远命不中
- 如果放在 conversation 前缀 → 不影响 system_instruction 的 cache → 但联想内容自身不被缓存

**你能接受"联想功能开启时，主模型的 KV Cache 命中率可能下降"吗？**

> 这里有个折中思路：联想内容走 conversation 通道而非 system_instruction，
> 这样 system_instruction（人格+工具）的缓存依然稳定。
> 你觉得这个折中方案可以吗？
