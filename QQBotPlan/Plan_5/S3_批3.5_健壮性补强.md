## 主题 A — 压缩 merge-save 用陈旧快照覆盖 metadata（最严重，根因单一）

**根因**：`compress_if_needed` 的 merge-save（checkpoint.py:1604-1637）。压缩期间 flash_lite 网络调用（秒级，只持 compress_lock，不持业务 _get_lock）让出事件循环，并发 append/flush_buffer 经唯一取号入口推进了**磁盘** metadata.next_round_id/next_step_id/generation 并写 state.json。压缩结束后 save 写盘的是 Phase1 带入的**陈旧 t_file**，merge 段只 `max(total_messages_ever)`（L1626），号源/generation/record_state 全沿用陈旧值 → 磁盘号源**回退** → 下次 append 重发已用过的 round_id/step_id → 违反「round_id 全局单调绝不复用」铁律，且 next_round_id 只存 T 文件 metadata、无 state 备份，**不可自愈**。

| finding | 严重度 | 说明 |
|---|---|---|
| compress-clobbers-numbersource | **critical** | next_round_id/next_step_id 回退 → round_id 复用 |
| compress-metadata-clobber | critical→high | 同根因（并发维度复述）|
| compress-generation-regression | high | generation 回退，T<state 永久不一致，恢复只抹 generation 不修号源 |

**修复**（一处）：merge-save 段（L1622-1637）落盘前，对 `next_round_id / next_step_id / generation` 也从 `current_t_file.metadata` 取 max（与 total_messages_ever 同处理），record_state.last_compressed_round_id 同样取较大。即「号源/世代只进不退」。
**验收**：单测——模拟压缩快照 next_round_id=45，压缩期间磁盘推进到 50，merge-save 后磁盘必须仍是 ≥50。

## 主题 B — 崩溃恢复 buffer/WAL 边界（一致性重构）

| finding | 严重度 | 根因 | 修复方向 |
|---|---|---|---|
| numbering-1 / first-touch-absorb-no-dedup | high | buffer_message first_touch 吸收残留 WAL 无去重 + 抢先 `_recovered.add` 永久屏蔽 _do_recover（docstring 谎称有去重，代码无）| first_touch **不自行吸收**残留 WAL，改为触发 `_recover_window_if_needed`（统一走带去重的 _do_recover）；或吸收时对 T 文件 existing_ids + _wal_key 双重去重，且**不抢标 _recovered** |
| residual-absorbed-not-rewal | high | first_touch 吸收残留进内存后未重写新 WAL，二次崩溃丢 | 若保留吸收路径：吸收后按新 seq 重新 wal_append 再清旧 WAL |
| flush-empty-wipes-residual-wal | **critical(未验)** | flush_buffer 空 buffer 分支（L753-756）无条件 wal_clear，未 replay 即删残留 WAL → 崩前未落盘消息永久丢 | 空 buffer 时**不删残留 WAL**；若有残留先 replay/触发恢复 |
| numbering-2 / flush-walclear-race-dataloss | high→medium | flush_buffer 锁外 pop pending（L753）后 await 让出，并发 buffer_message 写入新 WAL 行，flush 结束 wal_clear 删整文件（含新行）→ 新消息留内存无 WAL | wal_clear 精确化（只删本次 flush 的 seq 范围）；或 pop pending 移进锁内消除并发窗口 |
| replay-dedup-ignores-wal-key | high(未验) | _do_recover replay 只按 message_id 去重（L928-931），message_id=None（user/assistant 补录常无）恒不去重 → 重复落盘 | replay 去重改用 WAL 行的 `_wal_key`（{window}#seq{N}），覆盖 message_id=None |
| numbering-3 | high | T 文件损坏 → _create_empty 重置 next_round_id=1，但 state.json 独立存活（current_round_id=r000050）→ 重号 + assign_round 沿用旧轮号源从 1 起 | T 损坏重建时，从 state.json + 现存 .corrupt messages 的 max round_id/step_id 推导 next_round_id/next_step_id |

**注**：缓冲的 user 消息本就不带 message_id（router_mixin 339-348/404-413），故 message_id 去重对它们普遍失效——主题 B 的去重应统一以 `_wal_key` 为准。
