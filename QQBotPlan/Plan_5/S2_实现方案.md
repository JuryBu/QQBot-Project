## 二、三处断链 · 当前真相（S1 后）

### ① 主模型选型断链
- 运行时实际调用模型 **只由 provider 顶层 `model` 决定**：`openai_source.py:395`(`model=provider_config.get("model")`→`set_model`) → `:867`(`model=model or self.get_model()`) → `:869`(payload)。
- 面板「主模型选型」读写的是 `model_config.model`（`models.py` `update_main_model` L133-134 / `get_main_model` L102）。
- 当前 `cmd_config.json` 不一致：顶层 `model=gemini-2.5-flash`（生效）、`model_config.model=gemini-3-flash-preview`（仅回显）。
- `model_config` 在框架内唯一读取者是 `migra_helper`，因 provider 已有 `provider_source_id` 被 L72 `continue` 跳过 → **彻底死字段**，框架不会自动同步到顶层。

### ② FlashLite 整块文件断链
- 面板写 `AstrBot/data/plugins/astrbot_plugin_flashlite/config.json`（B 文件，`models.py:14 FLASHLITE_CONFIG`）。
- 运行时 `self._raw_config` 来自框架按 `_conf_schema.json` 注入的 `AstrBot/data/config/astrbot_plugin_flashlite_config.json`（A 文件，仅 11 键，`star_manager.py:822-826`）。
- **新发现1**：`storage_policy` 实际生效——`persistence/main.py:752-762 _load_storage_policy` 直接读 B 文件，与面板双源读写同一物理文件，绕过 schema。旧文档「纯空写无人读」已过时。
- **新发现2（BOM）**：框架写 A 文件用 `utf-8-sig`（带 BOM，`astrbot_config.py:49/161`），面板用 `utf-8`（`models.py:31/37`）。直接改面板文件指向会 `JSONDecodeError` / BOM 反复横跳。
- **新发现3（热加载假象）**：所有阈值在 `__init__` 一次性固化成实例属性（`self._sync_interval` L116 等），`_raw_config` 是启动时对象引用，改文件不重读。

### ③ Sandbox 路径分裂
- 统一根 = 项目根 `Sandbox/`（含 base_tools/config/workspace）；`tool_registry.SANDBOX_ROOT`(L24) / `sandbox.py`(L27) 上溯 4 层算法。
- S1 已修 `main.py:1840-1844`（base_tools）一处。**剩余错误路径**（仍指向插件内 Sandbox）：
  - `main.py:2069` `_get_tool_timeout_s` 读 base_tools → 插件内不存在 → FileNotFoundError → 静默 30s 兜底，所有工具自定义 timeout 失效。
  - `main.py:3273` env.json 注入 → 读插件内陈旧副本（schema 与真文件已不同）→ 给主模型注入过时环境信息。
  - `main.py:251` CostTracker init → cost_logs 写插件内（有 2 个历史 json）。
  - 兜底 fallback：`main.py:5647` / `4589`(send_image) / `4627`(send_file) 三处 `else` 在 `self._sandbox is None` 时退回插件内路径。
- 根因：`main.py` 顶部（约 L13）只有 `_PLUGIN_DIR`，无 `SANDBOX_ROOT` 常量，各处自建发散路径。

## 三、修复点清单（6 组）

### 组 1 — 主模型选型（文件：`BossLady_Console/backend/routers/models.py` + `cmd_config.json`）
1. `update_main_model` L131-144：写 `model_config` 的同时**同步写顶层** `p['model']=req.model`；max_tokens/temperature/thinking 同步写 `p['custom_extra_body']`（顶层 `custom_extra_body` 现有 `temperature:0.8`，是生效路径——**实现前先核对 openai_source 实际读取字段**）。
2. `get_main_model` L96-111：回显改为**优先读顶层** `p.get('model')`，反映真实运行值。

4. 框架侧 `openai_source.py` **不动**（正确，改它偏离上游）。

### 组 2 — 文件断链 + schema 批1（**三件必须同批**）
1. `_conf_schema.json` 扩声明【批1 现存断链键】：
   - `model`(string, 默认 `gemini-3.1-flash-lite-preview`)
   - `tool_model`(object：`model`/string、`thinking_budget`/int=2048、`api_keys`/list)
   - `image_model`(object：`model`/string=`gemini-2.5-flash-image`)

   - cost_tracker 拆为 `cost_usd_to_cny`(float=7.2) + `cost_custom_pricing`(string,JSON)（custom_pricing 是动态模型名子键，**不可 object 展开**，否则被框架递归删）→ 同步改 `main.py:252-256` 读取逻辑。
2. `models.py:14` `FLASHLITE_CONFIG` 指向改 A 文件 `AstrBot/data/config/astrbot_plugin_flashlite_config.json`。
3. `models.py:31/37` `_load/_save_flashlite_config` 编码 `utf-8` → `utf-8-sig`。
4. `models.py:214` 键名对齐：`sync_interval` → `sync_trigger_interval`（与 schema 一致）。
- ⚠️ storage_policy：保留 persistence/面板走 B 文件特例。**迁移 FLASHLITE_CONFIG 后**，面板 storage_policy 端点（`models.py:289-334`）需单独保留指向 B 文件，或确认其读写路径不随 FLASHLITE_CONFIG 常量变动。实现时核对。

### 组 3 — 动态键 json.loads（文件：`main.py`）

### 组 4 — Sandbox 路径统一（文件：`main.py`）

## 七、前置验证结论（改法钉死，wf_3eb5ec5a）

### V1 主模型参数
- temperature/max_tokens **唯一生效路径 = provider `custom_extra_body`**（`openai_source.py:462-474`），model_config 死字段。
- **thinking 雷区**：openai 兼容层对 Gemini thinking 无转换（`openai_source.py:406-418` 仅处理 ollama），裸 `thinking_level/budget` 不生效；需嵌套 `custom_extra_body.google.thinking_config` 或 `reasoning_effort`，且要对端点实测。**S2 暂不动 thinking**（配了不生效反误导），记后续实测再加。
- 组1 最终：`update_main_model` 写顶层 `model` + `custom_extra_body.{temperature,max_tokens}`；`get_main_model` 读顶层+custom_extra_body；清理 model_config 死字段；thinking 字段 S2 跳过。

### V2 主模型 provider 厘清（重要）
- `gemini_proxy` 是**幽灵 provider**：cmd_config 无该 id（grep proxy 仅命中 http_proxy）。sp.preferences 多条会话 `provider_perf_chat_completion='gemini_proxy'` 是历史遗留脏数据。
- `get_using_provider`(`manager.py:228-271`)：sp 指定 gemini_proxy → inst_map 找不到 → 回落 `default_provider_id=gemini_pro`（每次 warning L267-269）。
- **结论**：运行时会话实际回落用 **gemini_pro**；面板 get/update 也操作 gemini_pro（第一个 enable=true，=default）。**全链路一致**，改 gemini_pro 顶层 model 即正确生效。
- 顺手：清理 sp `provider_perf_chat_completion='gemini_proxy'` 脏数据消 warning（主控做，data_v4.db）。
- /model 不持久化（纯内存），重启回落 provider_config model；preferences 无 model 键。

### V3 三点
- **SANDBOX_ROOT**：`from .tool_registry import SANDBOX_ROOT` 安全（无回指 import），值=项目根 Sandbox。组4 用它不自建。
- **env.json**：旧副本(description/os/python_version/network) 与真文件(sandbox_version/total_storage_mb/ram_limit_mb/exec_timeout_*/tool_count) **字段集完全不同**。组4 切真文件**必须重写注入文案** `main.py:3280-3289`：改读 sandbox_version/total_storage_mb/ram_limit_mb/exec_timeout_default_ms(÷1000)/exec_timeout_max_ms(÷1000)/tool_count；删 os/python_version/network；硬编码 30s/300s/内存描述改为真文件动态生成。
- **定价表**：两目标模型均精确命中 cost_tracker PRICING，**不动 cost_tracker**。

### 范围调整
- **组6 reload 后置**：先调研 AstrBot 插件配置 reload 机制再定，S2 第一轮先落组1-5（核心断链），reload 单列。
- cmd_config 存量对齐 + 历史 json 迁移 + sp 脏数据清理：主控亲自做。
