目前我下面所有内容假设是我们能获取的QQ数据结构是每个群聊和个人对话窗口像一本本书一样，接收到就会存在我们的内存里，实际不会删除，然后QQbot可以时刻翻书看原内容，这里的假设是存入本地，不会自己删除，如果错误请你指出

根据此，我们设计两层式对话管理机制

一层是内存实际存储的对话C，这份对话不会自己删除，是所有群聊所有窗口所有信息的集合，我们需要自己管理

一层是实际发送的请求体C‘，包含了每个窗口的CHECKPOINT压缩内容，记忆，设定，工具说明，没有压缩的当前新传输上下文或者工具调用请求结果等

我们需要设立一个东西来维护，设计如下

对于各个窗口，设立一个CHECKPOINT上下文消息上限，按照token估计来计算，图片什么的也等效，达到CHECKPOINT就对当前窗口CHECKPOINT之前的消息调用LITE模型进行压缩，压缩率在10%-35%，同时进行压缩后主动的记忆存储和Knowledge更新操作，之后的这个窗口的实际发送的请求体C‘就变成了CHECKPOINT压缩+未压缩内容(大概最近十条左右不压缩，即使这样加起来超过CP触发上限)，之后要是继续触发就继续这样，维持高效的对话，实际上是每个窗口的RNN级别的对话，需要注意，这里的消息包括模型自己在当前窗口工作，任务的记忆也在其中，不是单纯的QQ信息组合体

### 问题分析

| 问题 | 当前 | 理想 |
|---|---|---|
| 淘汰策略 | FIFO 20条硬上限 | 分类淘汰——核心事实不淘汰 |
| 存储结构 | 纯文本列表 `["生日8月12日", ...]` | 结构化条目 `{summary, content, category}` |
| 注入粒度 | 全灌或不灌 | 默认只灌 summary，需要时才拉 content |

### 提议的分层设计

**每条 Fact 结构：**
```json
{
  "summary": "大三AI专业学生",
  "content": "用户提到自己是南师大大三AI专业的，对DL和GenAI特别感兴趣",
  "category": "pinned",
  "created_at": "2026-03-01",
  "last_referenced": "2026-04-04"
}
```

**三种分类：**
- **`pinned`** — 身份/性格/固定信息（生日、专业、性格特点） → **永不淘汰**，上限 10 条
- **`dynamic`** — 近期状态/偏好/正在做的事 → **按 last_referenced 淘汰**，上限 15 条
- **`archived`** — 淘汰的 dynamic 不删除，移入归档 → 主动查询可恢复，上限 30 条

**注入策略：**
- 默认注入（FlashLite/inject 钩子）：只拼 [summary](cci:1://file:///c:/Users/<user>/Desktop/AstrBotLauncher-0.1.5.6/AstrBot/data/plugins/astrbot_plugin_flashlite/main.py:2298:4-2367:39) 列表（很短，几百字）

- [knowledge_card(qq_id, detail=true)](cci:1://file:///c:/Users/<user>/Desktop/AstrBotLauncher-0.1.5.6/AstrBot/data/plugins/astrbot_plugin_flashlite/main.py:1606:4-1623:33) ：连 archived 也返回

**FlashLite 标记行升级：**
```
PROFILE_UPDATE=QQ号:category:summary|content
```
比如：`PROFILE_UPDATE=123456789:pinned:大三AI专业|用户提到自己是南师大大三AI的学生`
