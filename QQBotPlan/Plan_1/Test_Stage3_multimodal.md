# 🧪 Stage 3 测试：多模态视觉处理

> 对应: Plan_1.md Stage 3 | 前置: Stage 1-2 已完成

---

## 测试目标
验证 context_enhancer 的图片描述功能在实际群聊中的端到端工作链路。

---

## 场景 1：单张图片消息

### 模拟输入
```
[群聊 <GROUP_B>]
14:30:01 古希腊掌管铯欲的神: @老板娘 你看这个好吃不
14:30:01 [图片: 一盘红烧肉的照片]
```

### 预期行为
1. context_enhancer 拦截图片，调用 Flash Lite 生图描述
2. 描述结果类似：「[Image: 一盘色泽红亮的红烧肉，摆在白色瓷盘中]」
3. 主模型收到文字描述 + 原始 prompt，以老板娘人格正常回复
4. 响应延迟 ≤ 3 秒（含图描时间）

### 验证方式
- [ ] 检查 AstrBot 日志中 image_caption 调用成功
- [ ] 验证主模型收到的 prompt 包含图片描述文本
- [ ] 验证回复内容与图片相关（不是答非所问）

## 场景 2：连续多张图片（表情包轰炸）

### 模拟输入
```
[群聊 <GROUP_B>]
14:31:00 Jury_鸽姬布: 我去，看看这些表情包
14:31:01 [图片: 滑稽.jpg]
14:31:02 [图片: 狗头.jpg]
14:31:03 [图片: 哭笑不得.jpg]
14:31:04 [图片: doge.jpg]
14:31:05 [图片: 流泪猫猫头.jpg]
14:31:06 Jury_鸽姬布: @老板娘 你觉得哪个好笑
```

### 预期行为
1. 5 张图片不应全部调用描述（节省 token）
2. 策略：保留最后 2 张原图/描述，其余标记为「[表情包 x3]」
3. max_images_in_context=4 的配置生效
4. 不应因图片过多导致请求超时

### 验证方式
- [ ] 日志显示图片数量被限制
- [ ] 主模型收到的上下文中只包含有限数量的图片描述
- [ ] 总处理时间 ≤ 5 秒

## 场景 3：QQ 聊天记录截图（转发消息）

### 模拟输入
```
[群聊 <GROUP_B>]
14:35:00 张三: 转发了一个聊天记录
14:35:00 [转发消息聊天记录]:
    李四: 今天数据库课考的怎么样
    王五: 寄了，完全没复习
    李四: 我也是啊哈哈哈
    王五: 期末稳了（确信
14:35:05 张三: @老板娘 老板娘你看看这群人考得怎么样
```

### 预期行为
1. 检测到转发消息结构 → 提取为结构化文本
2. 不需要图片描述（转发消息是文本组件）
3. 主模型看到完整的转发聊天内容
4. 回复应包含对考试话题的理解

### 验证方式
- [ ] 转发消息被正确提取为文本格式
- [ ] 主模型的回复涉及考试相关内容
- [ ] 未触发不必要的图片描述调用

## 场景 4：图片+长文本混合

### 模拟输入
```
[群聊 <GROUP_B>]
14:40:00 学术帝: @老板娘 帮我看看这个代码有什么问题
14:40:00 [图片: 一张 IDE 截图，显示 Python 代码有红色错误标记]
14:40:01 学术帝: 我跑了一下午了一直报 TypeError，说什么 NoneType has no attribute 'split'
14:40:02 学术帝: 是不是这里的 response 变量没赋值？
```

### 预期行为
1. 图片被 Flash Lite 描述为代码截图内容
2. 文本上下文和图片描述合并为完整 prompt
3. 主模型能结合图片描述+文字描述给出有用回复
4. 理想情况：识别出 NoneType 错误的常见原因

### 验证方式
- [ ] 图片描述包含代码相关内容
- [ ] 回复质量：提到可能的 None 值问题
- [ ] 上下文增强的 prompt 包含图片描述+用户文本

## 压力测试

### 图片风暴测试
- 30 秒内连续发送 20 张图片
- 验证：系统不崩溃，不超时，图片描述队列正常消化
- 预期：只处理最近 max_images_in_context 张

### 大图测试
- 发送 4K 分辨率图片
- 验证：Flash Lite 的图片描述不因分辨率过高而超时
- 预期：自动缩放或超时降级为「[大图]」
