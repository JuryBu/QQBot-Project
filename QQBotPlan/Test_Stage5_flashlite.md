# 🧪 Stage 5 测试：Flash Lite 中断引擎

> 对应: Plan_1_models.md Flash Lite 部分 | 前置: Stage 4 消息持久化

---

## 测试目标
验证 Flash Lite 作为"CPU 中断"的核心调度功能：同步/异步触发、Knowledge 更新、响应判断。

---

## 场景 1：同步触发（每 5 条群消息触发一次）

### 模拟输入
```
[群聊 <GROUP_B> - 普通闲聊，老板娘未被提及]
16:00:00 张三: 今天中午吃啥
16:00:05 李四: 我去食堂二楼了
16:00:10 王五: 二楼那个拉面不行，太油了
16:00:15 赵六: 一楼的黄焖鸡还行
16:00:20 钱七: 我在外面点的外卖
→ [触发 Flash Lite 第 1 次同步调用]
16:00:25 张三: 外卖什么
16:00:30 钱七: 麻辣烫
16:00:35 李四: 我也想吃
16:00:40 王五: 明天一起去
16:00:45 赵六: 好
→ [触发 Flash Lite 第 2 次同步调用]
```

### 预期行为
1. **第 1 次触发**：
   - Knowledge 更新：「<GROUP_B> 群最近在讨论午餐选择」
   - 响应判断：与老板娘无关 → **不触发主模型** → 静默返回
2. **第 2 次触发**：
   - Knowledge 更新：「继续讨论午餐，有人提议明天一起去吃麻辣烫」
   - 响应判断：仍与老板娘无关 → **不触发主模型**

### 验证方式
- [ ] Flash Lite 被调用了 2 次
- [ ] Knowledge 中 <GROUP_B> 的 summary 被更新了 2 次
- [ ] 主模型未被触发（无回复发出）
- [ ] Flash Lite 每次调用延迟 ≤ 2 秒

---

## 场景 2：异步触发（@老板娘）

### 模拟输入
```
[群聊 <GROUP_B>]
16:05:00 张三: 今天中午吃啥
16:05:05 李四: 我去食堂二楼了
16:05:10 王五: @老板娘 你觉得吃啥
→ [立即异步触发 Flash Lite]
```

### 预期行为
1. 第 3 条消息 @老板娘 → **立即**异步触发 Flash Lite
2. 不等到累积 5 条才触发
3. Flash Lite 判断：明确 @ 事件 → **强制通知主模型响应**
4. 主模型收到：Knowledge + 上下文 + @消息 → 回复

### 验证方式
- [ ] Flash Lite 在 @消息到达后 500ms 内被触发
- [ ] Flash Lite 返回"触发主模型"指令
- [ ] 主模型收到的上下文包含前 2 条背景消息
- [ ] 主模型正常生成回复

2. Flash Lite 判断：关键词+语义分析 → 这是在和老板娘说话 → 触发主模型
3. 主模型以老板娘人格回复

## 场景 4：语义判断——不该回复的场景

### 模拟输入
```
[群聊 <GROUP_B> - 进入第 5 条触发]
16:15:00 张三: 我今天去面试了
16:15:05 李四: 哦？怎么样
16:15:10 张三: 面试官问我什么是KV Cache
16:15:15 李四: 哈哈哈这不是老板娘在用的那个吗
16:15:20 王五: 说起来还有个叫老板娘的奶茶店
→ [同步触发 Flash Lite]
```

### 预期行为
1. 消息中提到"老板娘"，但是**在谈论别的事**
2. Flash Lite 语义分析："老板娘"出现在引用/比喻中，不是在和老板娘说话
3. 判断结果：**不触发主模型**
4. Knowledge 更新：「群里在讨论面试经历和KV Cache技术」

### 验证方式
- [ ] Flash Lite 返回"不需要触发"
- [ ] 主模型未被调用
- [ ] Knowledge 正确更新了话题摘要

## 场景 6：CHECKPOINT 触发

### 模拟输入
```
[群聊 <GROUP_B> - 累积了大量消息，token 估计接近上限]
...（前 200 条消息，约 50000 token）...
16:30:00 第 201 条消息到达
→ [token 估计超过 CHECKPOINT 上限]
→ [异步触发 Flash Lite：CHECKPOINT 压缩事件]
```

### 预期行为
1. Flash Lite 检测到 token 超限
2. 对前 190 条消息进行 10-35% 压缩
3. 最近 ~10 条保持不压缩
4. 压缩结果写入 checkpoint_history 表
5. Knowledge 同步更新

### 验证方式
- [ ] 压缩率在 10-35% 范围内
- [ ] 压缩后的摘要保留了关键信息（人名、话题、结论）
- [ ] 最近 10 条消息原文未受影响
- [ ] Knowledge 中本窗口的 summary 已更新

## 压力测试

### 高频同步触发
- 模拟活跃群：每秒 3 条消息 → 每 1.7 秒触发一次 Flash Lite
- 持续 5 分钟 → 约 180 次 Flash Lite 调用
- 验证：无请求堆积、无内存泄漏、Knowledge 更新一致

### Flash Lite 延迟基线
| 操作 | 预期延迟 | 上限 |
|------|----------|------|
| 语义判断（不压缩） | ≤ 800ms | 1.5s |
| 语义判断 + Knowledge 更新 | ≤ 1.2s | 2s |
| CHECKPOINT 压缩（200条→70条） | ≤ 3s | 5s |
| CHECKPOINT 压缩 + Knowledge + Memory 建议 | ≤ 4s | 6s |

### API 开销估算
- 5 条/次 × Flash Lite 输入 ~2000 token + 输出 ~500 token
- 活跃群 300 条/小时 → 60 次 Flash Lite/小时
- 每小时 Flash Lite 开销 ≈ 150K token
- gemini-3.1-flash-lite 定价：约 $0.015/小时（极低）
